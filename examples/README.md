# Examples

This directory contains **illustrative example materials** showing how the repository is structured and what kinds of outputs are generated when the toolkit is run for a single game.

The purpose of this folder is **documentation and orientation**. It is not intended to duplicate the full repository output for every processed game.

## What this directory contains

The `examples/` folder may include:

- example game configuration files;
- selected sample outputs from one fully processed game;
- reduced or partial datasets intended only to illustrate file structure and naming conventions;
- selected tables, figures, and control artefacts produced by the pipeline.

## What this directory does not aim to provide

This folder does **not** necessarily include:

- the full raw corpus;
- every chunk file generated during collection;
- every output table generated by every analysis stage;
- every Excel workbook or every visual output;
- or a complete archival copy of a processed game dataset.

Instead, it provides a **compact, representative example** of the workflow.

## Recommended use

Use this folder to understand:

- how a game configuration file is structured;
- how outputs are organised by stage;
- what kinds of files are generated in `raw`, `processed`, `tables`, and `figures`;
- how validation and pipeline control outputs look;
- and how naming conventions are applied across the repository.

## Suggested structure

A typical `examples/` structure may look like this:

```text
examples/
├── README.md
├── config/
├── sample_outputs/
│   └── <example_game_slug>/
│       ├── raw/
│       ├── processed/
│       ├── tables/
│       └── figures/
└── dictionaries/
```

## `config/`

This folder may contain:

- a generic configuration file such as `example_game.json`;
- one or more real game configuration files used in the repository.

These files illustrate how to define:

- game identifiers;
- collection parameters;
- emotion-analysis parameters;
- theme-analysis parameters;
- word-cloud parameters.

## `sample_outputs/`

This folder contains **selected outputs** from one example game.

The purpose of these files is to show:

- expected filenames;
- directory structure;
- table formats;
- figure workbook formats;
- validation outputs;
- and pipeline-control artefacts.

Depending on repository needs, some files may be reduced samples rather than full outputs.

## `sample_outputs/<example_game_slug>/raw/`

This section may include examples of:

- combined raw review exports;
- metadata files such as progress summaries.

These files illustrate the output of the collection stage.

## `sample_outputs/<example_game_slug>/processed/`

This section may include examples of:

- cleaned datasets;
- enriched datasets from emotion analysis;
- enriched datasets from theme analysis;
- metrics JSON files;
- validation summaries;
- pipeline summaries;
- output manifests.

These files illustrate the outputs of preparation, enrichment, validation, and orchestration stages.

## `sample_outputs/<example_game_slug>/tables/`

This section may include selected CSV tables from:

- basic metrics;
- temporal metrics;
- text metrics;
- emotion metrics;
- theme metrics.

These files illustrate the main analytical outputs used for interpretation.

## `sample_outputs/<example_game_slug>/figures/`

This section may include:

- selected Excel figure workbooks;
- selected word-cloud images;
- word-cloud frequency tables.

These files illustrate the visual-output stages of the repository.

## Relationship to the main repository structure

The main repository is designed to generate outputs under:

- `data/raw/steam_reviews/<game_slug>/`
- `data/processed/steam_reviews/<game_slug>/`
- `results/<game_slug>/tables/`
- `results/<game_slug>/figures/`

The `examples/` directory provides a **small, curated subset** of that broader structure.

## Notes on sample files

Where sample files are used, they may:

- contain only a subset of rows;
- contain only a subset of generated outputs;
- be renamed with a suffix such as `_sample`;
- be included for demonstration rather than for substantive analysis.

## Good practice

When updating this directory, it is recommended to:

- keep the example compact;
- include only representative files;
- avoid unnecessarily large raw exports;
- preserve the repository naming conventions;
- and ensure that example files remain consistent with the current workflow.
